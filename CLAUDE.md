## State — always maintain

BUILD_STATUS → `BUILD_STATUS.md`. Every story that lands updates its row — **status + spec link, and nothing else**: no SHAs, no branch names, no summary of what the story proved. The tickets carry the detail and git answers "where does it live"; anything duplicated in the tracker is a fact that rots. **Update the row before the merge, in the last commit on the feature branch** — updating after means committing straight to `dev`, which this workflow doesn't allow. Every change that wasn't a planned story gets a line in the off-epic ledger. Keeping BUILD_STATUS current is **non-negotiable** — a stale tracker misleads the next session with authority, which is worse than no tracker. Anything raised along the way (idea, issue, deferred feature) is captured in the epic map's backlog — that is its one home. Regression checklist → `REGRESSION_CHECKLIST.md` (a bug that escapes to a human adds a line; a recurring line graduates into the Maestro smoke suite).

## Hard rules (this codebase)

- **All workspace access goes through the authorization guard; never query workspace tables directly.** Workspace-scoped service methods take the resolved `Membership` as a parameter (Artifact 03). No inline authority checks anywhere.
- **All capability gating goes through the entitlement service** — `can(traveler, capability)`; never inline tier checks (ADR-009). v1: it returns full access.
- **The ledger is append-only.** Corrections, waivers, settlements, reassignments are new Transfer entries — never UPDATE or DELETE on ledger rows (INV-8). Splits must sum to the expense total transactionally (INV-7).
- **API changes are additive only within /v1** — never rename, retype, remove, or change semantics of anything shipped (ADR-008; old app versions live for weeks).
- **Modules reference each other by ID + service interface only** — never another module's tables or internals (ADR-002).
- **Mobile: no raw fetch/API calls in UI code** — everything through the repository/local-cache layer's typed apiClient (ADR-001, P6).
- **Never log secrets, tokens, passwords, raw payloads, or PII.** Reference entities by ID (P3).

## Stop rules — ask the owner before doing

Touching the authorization guard or isolation semantics · schema migrations beyond the current story's additive tables · anything in auth/token handling · changing ledger semantics · changing entitlement/gating semantics · deleting data or writing destructive migrations · changing publish/visibility semantics · anything that breaks /v1 additivity · anything irreversible.

## Never commit secrets

No `.env` or any file containing credentials, keys, tokens, passwords, or **PII** enters a commit — ever. This is P3 (never-log-secrets) extended from logs to commits: secrets never enter *any* durable artifact. Two layers, no third-party tool required:

- **Structural guarantee (the real defence): `.gitignore` the secret files before the first commit.** `.env`, `.env.local`, anything holding credentials — gitignored so they *cannot* be staged. Real secrets live only in gitignored files locally and in the platform's env-var UI in prod; the repo carries `.env.example` with placeholders only. If nothing sensitive is ever tracked, there is nothing to leak — it does not depend on anyone remembering.
- **Backstop (soft): scan the staged diff before every commit.** Check the staged content for secret/PII patterns (`API_KEY=`, `SECRET`, `PASSWORD=`, tokens, long high-entropy strings, any `.env` in the staged files) and refuse to commit on a match. It is a backstop *because* it's an instruction — lead with the gitignore, never let the scan become an excuse to relax it.

A committed secret is compromised even if a later commit deletes it (it stays in history) — **rotate it.**

## Git workflow (promotion pipeline)

Branch→environment mapping: `main` → **production** (protected base, real data) · `preprod` → a **pristine copy of production** (the verify gate; data mirrors prod) · `dev` → **long-lived shared preview** (persistent; reseed at the developer's discretion, *not* ephemeral) · **local** → the **ephemeral full-stack instance** (fresh DB every redeploy) where feature branches are built and tested.

Flow: feature branches (`feature/<story-id>-<slug>`) are tested on the **local ephemeral instance**, then **squash-merged into `dev`**. Promotion: **cherry-pick** the verified changes from `dev` → `preprod`, verify against the prod-copy, then promote `preprod` → `main`. Commit convention: `feat(scope): <story-id> …` / `fix(scope): <story-id> …` — **the story id in the message is mandatory**: it is how commits are located (`git log --grep <story-id>`), since BUILD_STATUS stores no SHAs. *Commit messages carry **no agent signature** — no `Co-Authored-By`, no "Generated with" trailer, no attribution line; just the conventional message.*

**Promotions are propose-first.** The agent may run routine git directly — create the feature branch, commit to it — but it **proposes and waits for approval before any promotion**: squash-merging `feature → dev`, promoting `dev → preprod`, or `preprod → main`. Promotions cross environments and are hard to reverse; each is a checkpoint, not an autonomous step.

**The promotion unit is the epic, and each hop has its own mechanic** *(founder decision, 2026-07-17, at Epic 0's promotion — the first one this repo ever ran)*:

- **`dev → preprod` is a SQUASH**, one commit per promotion, named for what it ships (`feat(skeleton): Epic 0 — the walking skeleton, dev-verified`). **Why:** an epic is the shipping unit, so it is the rollback unit — when a deployment misbehaves you want *one* SHA to revert, not a boundary to locate under pressure. A partial revert of an epic is rarely coherent anyway (S0.6 without S0.5's tri-state does not build). The cost is accepted knowingly: **preprod/main carry no per-story history**, so `git log --grep <story-id>` works on **`dev` only** — which is where the granularity is wanted, for investigation rather than recovery.
- **`preprod → main` is a FAST-FORWARD.** Never squash or cherry-pick this hop: it would mint a *different* SHA for identical content, destroying the property the pipeline exists for — that **prod carries exactly what preprod verified**, checkable with `git rev-parse main preprod`.
- **Consequence, permanent:** after the first squash, `dev` and `preprod` share no commits. Later promotions are therefore squash-or-cherry-pick and **never** fast-forward — and this is exactly where the footgun above lives: a cherry-picked subset can leave a dependency behind, so prefer promoting the whole epic.

**Footgun to watch:** cherry-picking after squash-merges can pull a change whose dependency you didn't pick — a subset that built in `dev` may not build in `preprod`. Track inter-change dependencies before promoting. **The mobile train is separate** (Artifact 04, ADR-010 — Android-only until the iOS activation): emulator/dev-build (local, non-EAS) → local backend · Play internal testing → preprod · Play release → prod. Release builds are made locally (Expo prebuild + Gradle) and uploaded manually.

## Verify on the local full stack — it is where feature branches are tested

**This is Artifact 04's environment topology (04 §83–89), restated here because S0.5 lost it:** feature branches are tested on the **full-stack local Docker instance** — `docker compose up` (Spring + Postgres 18 + MinIO), **fresh DB every redeploy** — and *then* squash-merged to `dev`. `dev` is the long-lived shared **preview**, not a test rung; testing a branch against it writes junk into the environment founders look at, and lets pre-existing data flatter a result that a fresh DB would have failed. Ephemerality is a testing tool, so it lives where you test (04 §85). The mobile side points at it the same way: **emulator / dev-build → local Docker backend** (04 §89).

Two rungs above it, each proving something the one below cannot:

- **The web preview is verified by building and running `mobile/Dockerfile.web-preview`**, never `expo export` + a static server *(S0.5, owner call — it found a real bug on its first run)*. Only the container exercises the **true build path** (`npm ci` from the lockfile, export inside the image) and the **true server** (Caddy + `Caddyfile.web-preview` — SPA `try_files`, cache headers, security headers). Anything else proves the bundle and nothing about what Railway serves; that gap hid a `Cache-Control` matcher that never fires for real URLs.
- **A sideload AC is the exception that proves the rule: it runs against the deployed `dev` backend on purpose** (S0.5) — the whole point is proving the *baked cloud URL*, which a local backend would defeat. Know which question the run is answering before choosing its backend.

The generalisation this repo keeps re-learning: **verify at the layer that ships, not the layer that is convenient.** Device ACs are closed on a device (S0.2's `getTokens()` was invisible to every green test); release-signed behaviour is proven on a release-signed build (S0.5's Google sign-in); server behaviour is proven on the real server.

## Running the local verification rig (the recipe, learned the hard way at S0.6)

**Every story that touches the mobile web surface or the native app runs this. It is written down because S0.6 rediscovered all of it from scratch — an hour of plumbing, none of it the story's actual work.** Ports are not arbitrary: the preview's port is *pinned* by two independent registrations, so change Metro's instead.

**The port map, and why it cannot be rearranged:**

| Port | What | Why it is fixed |
|---|---|---|
| 8080 | backend (docker compose) | `mobile/.env` and the container build arg point at it |
| **8081** | **preview container** | **Pinned twice**: the backend's dev CORS allowlist (`docker-compose.yml` `LARGATA_CORS_ALLOWED_ORIGINS` = 8081/8082 only) *and* the Google OAuth client's authorized JavaScript origins (console-registered — S0.6). Move the preview and Google refuses to render the button *and* the API preflights 403. |
| 8082 | Metro (`npx expo start --port 8082`) | The one that gives. Metro defaults to 8081 and *will* collide with the preview. |

```bash
# 1. Full stack (fresh DB every time — that is the point)
docker compose up -d --build
curl -s http://localhost:8080/v1/health          # {"status":"ok"}

# 2. Preview container — the TRUE build path (npm ci + export inside the image) and TRUE server
# `set -a; . ./.env; set +a` exports mobile/.env into the shell so the --build-arg values below
# resolve from it. Never paste the literals into the command: they land in shell history, and the
# transcript of whoever is watching. The file is gitignored; keep the values there.
cd mobile && set -a && . ./.env && set +a
docker build -f Dockerfile.web-preview \
  --build-arg EXPO_PUBLIC_API_BASE_URL="http://localhost:8080" \
  --build-arg EXPO_PUBLIC_FIREBASE_API_KEY="$EXPO_PUBLIC_FIREBASE_API_KEY" \
  --build-arg EXPO_PUBLIC_FIREBASE_AUTH_DOMAIN="$EXPO_PUBLIC_FIREBASE_AUTH_DOMAIN" \
  --build-arg EXPO_PUBLIC_FIREBASE_PROJECT_ID="$EXPO_PUBLIC_FIREBASE_PROJECT_ID" \
  --build-arg EXPO_PUBLIC_FIREBASE_APP_ID="$EXPO_PUBLIC_FIREBASE_APP_ID" \
  --build-arg EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID="$EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID" \
  -t largata-preview:local .
docker run -d --name largata-preview -p 8081:8080 -e PORT=8080 largata-preview:local

# 3. Native DEV build on the emulator (UI + flows; JS comes from Metro, debug-signed)
export ANDROID_HOME="$LOCALAPPDATA/Android/Sdk"      # REQUIRED — see gotcha below
export PATH="$ANDROID_HOME/platform-tools:$PATH"     # REQUIRED — else install fails after a green build
npx expo run:android                                 # or: gradlew app:assembleDebug + adb install
npx expo start --port 8082                           # NOT 8081 (preview owns it)

# 4. Native RELEASE build + sideload (the artifact that ships; JS bundled, real keystore)
#    Only for ACs that a debug build cannot close — see "which build proves what" below.
powershell -File mobile/scripts/build-release.ps1    # -ApiBaseUrl defaults to https://api-dev.largata.com
#   ^ prompts for the keystore password; echoes the baked API URL + signing cert in its summary.
#     Other params: -KeystorePath (default ~/keys/largata-release.keystore), -Alias (default 'largata')
adb install -r mobile/android/app/build/outputs/apk/release/*.apk
```

**Which build proves what — do not spend a release build on a question a dev build answers.**

| | Dev build (`expo run:android`) | Release build (`build-release.ps1`) |
|---|---|---|
| Signed with | debug key (SHA-1 `5e8f1606…`) | the real keystore (SHA-1 `ff05c732…`) |
| JS from | Metro, live | bundled into the APK (Hermes bytecode) |
| Proves | UI, flows, native module wiring | **release-signed behaviour** — Google sign-in in particular, since it is the one thing that differs by signing key (S0.5) |
| Cost | ~1 min rebuild | full prebuild + Gradle + a password prompt |

A debug build works forever while the release one is broken — that is the S0.4 ninja trap and the S0.5 SHA-1 trap both. **Any AC about the shipping artifact is closed on a release build, on a device** (S0.5's rule); everything else runs on the dev build.

**The traps, each of which cost real time at S0.6:**

- **`adb` is not on the Git Bash PATH and `ANDROID_HOME` is unset.** Gradle fails with *"SDK location not found"*, which reads like a broken toolchain but is a missing shell variable. Do **not** "fix" it by writing `android/local.properties` — that tree is generated and gitignored (CNG), so the edit vanishes at the next prebuild. Export `ANDROID_HOME` instead. Separately, `expo run:android` will **compile successfully and then fail at install** if `adb` is not on PATH — the error is a `spawn ENOENT` deep in an ADBServer stack trace, nowhere near the word "adb".
- **Git Bash rewrites device paths.** `adb push x /data/local/tmp/y` becomes `C:/Program Files/Git/data/local/tmp/y` and fails with `secure_mkdirs() failed`. Prefix with `MSYS_NO_PATHCONV=1`.
- **`adb reverse` can be listed and inert.** `adb reverse --list` showed the tunnel while it carried zero traffic (the adb daemon had died — `adb kill-server && adb start-server` reveals it with "daemon not running"). Test the far end, never the plumbing: `adb shell "echo -e 'GET /status HTTP/1.1\r\nHost: localhost\r\nConnection: close\r\n\r\n' | nc localhost 8081 | head -2"`. If the tunnel is dead, point the app at Metro directly via `10.0.2.2` (the emulator's host-loopback alias) by pushing RN's dev pref:
  `<map><string name="debug_http_host">10.0.2.2:8082</string></map>` → `/data/data/com.largata.app/shared_prefs/com.largata.app_preferences.xml` (via `adb push` + `run-as … cp`).
- **`INSTALL_FAILED_UPDATE_INCOMPATIBLE`** means the emulator holds a build signed with a different key (e.g. a release APK from sideload testing). `adb uninstall com.largata.app` first — emulator state is disposable.
- **Metro serves `expo-router/entry.bundle`, not `index.bundle`.** Probing `/index.bundle` returns 404 and looks like a broken Metro; it is a wrong probe. Check `/node_modules/expo-router/entry.bundle?platform=android&dev=true` → 200.
- **`expo start` in non-interactive mode silently skips its dev server on a port clash** ("Port 8082 is being used… Input is required… Skipping dev server"). It exits *zero*, and `/status` still answers from the *older* instance — so the port looks healthy while nothing is bundling.
- **Evidence, not vibes.** Web: **`node mobile/scripts/drive-preview.js [url] [--shot out.png]`** — real headless Chrome, committed at S0.6 *because S0.5 wrote this driver, threw it away, and S0.6 wrote it again*. It reports the page text (empty = the S0.4 white screen), whether Google's iframe rendered, whether a One Tap overlay appeared, and every console/page error. Native: logcat showing `SignInHubActivity` → GMS `SignInActivity` (the real picker) plus a screenshot (`adb shell screencap -p /sdcard/x.png && adb pull …`). **Neither is sufficient alone:** a sign-in that reaches My Trips can still have failed — check `Traveler provisioned` in the backend log and the DB going 0 → 1. **Routing is not a round-trip** (S0.5).

## Google Cloud console: the OAuth client the preview needs (S0.6)

**The web preview's Google button needs an *authorized JavaScript origin* per browser origin, registered on the OAuth client — and this is the first story that reaches past Firebase's abstraction into the Cloud project underneath it.** S0.2–S0.5 never needed the Cloud console because Firebase managed the client invisibly. Landmines, all hit at S0.6:

- **A Firebase project may not appear in the Cloud console's project picker until you enter it once from Firebase** (Firebase console → gear → *view on Google Cloud*). Before that it looks like the project does not exist — and the console will happily offer to *create* one, appending a suffix because your real project already owns the name. **S0.6 configured a consent screen and an OAuth client in a phantom `largata-dev-502703` before noticing.** Nothing would ever have worked, with no error saying why: the origins simply would not apply. **Identify projects by NUMBER, never name — the real one is `309534715609` (`largata-dev`).** Same principle as ADR-006's environment identity: the name is a label, the number is the identity.
- **The client to edit is `309534715609-7jjk1jq73igts…`** — the `client_type: 3` (web) entry in `mobile/google-services.json`, and the value of `EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID`. The project contains ~5 auto-created clients (3 web, 2 Android); **edit only that one, delete none of the others.** The Android ones carry the SHA-1s that make native sign-in work; the other web ones are Firebase-managed. Editing a sibling leaves Google returning `400` for *our* client with no error saying "wrong client".
- **Registered origins (dev):** `https://founders.largata.com`, `http://localhost:8081`, `http://localhost`. Bare host and host:port are **distinct entries**. Redirect URIs stay empty — GIS's button flow delivers the credential to a JS callback and never redirects.
- **Propagation is 5 minutes to a few hours.** Until it lands, the button silently does not render — clean console, no page errors — which reads exactly like a code bug. **The only trustworthy check is whether Google's iframe is in the DOM:** run `node mobile/scripts/drive-preview.js` (committed at S0.6 — real headless Chrome, no new dependency) and read *"Google-rendered iframes: 1"*. **Ignore `/gsi/button`'s status code entirely** — S0.6 first built a `curl` watcher on it (reported 400 forever while the button worked), then found it returns **400 even from a real browser while rendering fine**. It is not the origin signal it appears to be, in any client. A check that contradicts the running system is worse than no check; twice burned by the same one.
- **Per project, not per app:** consent screen, OAuth clients, and origins are all per Cloud project. `largata-prod` will need its own of each, and a different `EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID` — that work belongs to the parked prod-rung story.
- **Consent screen in "Testing" publishing status only admits accounts on its test-user list.** Add each founder's Gmail before a demo, or they hit access-denied and it reads as our bug.
- **`mobile/google-services.json` is a point-in-time snapshot.** It carries only the **debug** SHA-1 (`5e8f1606…`); the release key (`ff05c732…`, registered at S0.5) exists in the console but not in that file. The console is authoritative — do not diagnose from the JSON alone.

## Gotchas (grows during the build)

- **The two release trains need different JDKs on one machine: backend = Java 25, Android = Java 21.** `JAVA_HOME` points at JDK 25 (S0.1's backend requirement), and the Android Gradle Plugin does not support it — the build dies at `configureCMakeDebug` reporting only *"WARNING: A restricted method in java.lang.System has been called"*, a JDK warning masquerading as the whole error, which reads like a broken toolchain rather than a version mismatch. **Already fixed and should stay fixed:** `mobile/plugins/withAndroidJdk.js` pins `org.gradle.java.home` to Android Studio's bundled JBR at every prebuild (override with `LARGATA_ANDROID_JAVA_HOME`). Never "fix" this by editing `android/gradle.properties` — that directory is generated and gitignored, so the edit vanishes at the next prebuild and the inscrutable CMake error returns. Hit at S0.2.
- **The Firebase JS SDK cannot be reliably bundled into an Expo web export — the web preview uses the Identity Toolkit REST API instead (S0.4), and going back to the SDK will re-break it.** Symptom: `Uncaught Error: Component auth has not been registered yet` at module load, blank page, clean network tab (so it sends you hunting in the wrong place). Cause: the SDK registers its auth component through an import *side effect* Metro tree-shakes out of the web bundle, so `getAuth`/`initializeAuth` have nothing to initialize. **Every documented Metro workaround was tried at S0.4 and none fixed the registration** — `unstable_enablePackageExports=false` (the top community answer, expo/expo#36588), steering `unstable_conditionNames` to `browser` (this *did* swap Firebase's RN build for the browser build — verified by the `getReactNativePersistence` marker going present→gone — and the error **persisted**), `initializeAuth` over `getAuth`, `.cjs` sourceExts. Each fixed a real layer without fixing the registration. **The pivot that works: import nothing from `firebase/*`.** `mobile/src/auth/firebaseWebRest.ts` signs in with `fetch` to `identitytoolkit.googleapis.com/.../accounts:signInWithPassword?key=<apiKey>` and refreshes at `securetoken.googleapis.com`; the tokens are ordinary Firebase ID tokens the backend already trusts (ADR-006 unchanged). **Native is unaffected** — it uses `@react-native-firebase` (a native SDK), and the `.native/.web` fork means only the web files ever imported `firebase/*`; there is no `metro.config.js` and the `firebase` npm dep is gone. If someone reintroduces the JS SDK for the preview, expect the white screen back. **Debugging lesson that cost several cycles: keep the loop at the *bundle* (grep the exported JS for the failing call sites), never at browser reloads** — the browser render is the slowest possible test and can't tell you *why*. Still true regardless of SDK-vs-REST: read `EXPO_PUBLIC_*` by direct static access (below), and `web.output:"single"` in `app.json` is required or the export emits no root `index.html` and a static host shows a directory listing, not the app.
- **Browser preflight to a *secured* endpoint needs CORS wired into the Spring Security chain, not the MVC layer — else it 401s with no CORS header and the browser blocks the real request.** A browser sends a preflight `OPTIONS` before any request carrying an `Authorization` header. That OPTIONS has no token, so if CORS lives in a `WebMvcConfigurer` (which runs *after* Spring Security) the chain 401s the preflight before any CORS header is added, and the real `GET` never fires — `Response to preflight request doesn't pass access control check`. Same family as the S0.2 filter-ordering gotcha. **Fix:** `DevCorsConfig` provides a `CorsConfigurationSource` **bean** (dev profile only), and `SecurityConfig` wires it with `.cors(Customizer.withDefaults())`, so Spring Security's own CORS filter answers the preflight `200` before auth. Prod/preprod run no profile → no bean → `.cors()` finds no source → no CORS (the invariant `ProdCorsAbsentIT` guards). **Invisible to native** (no CORS) and to any CORS test that only hits the *public* `/v1/health` — the regression test must preflight a *secured* endpoint (`DevCorsEnabledIT.preflightToASecuredEndpointIsAllowedBeforeAuth`). Hit at S0.4 the first time the web preview called a secured route.
- **Expo inlines `EXPO_PUBLIC_*` on web by literal text substitution of `process.env.EXPO_PUBLIC_X` — a computed `process.env[name]` is invisible to it and silently ships unresolved.** There is no runtime environment in a web bundle; Expo bakes these vars at export time by find-and-replacing the *exact source text* `process.env.EXPO_PUBLIC_FOO`. Read one indirectly — `const v = process.env[name]`, a `required(name)` helper, destructuring, anything non-literal — and the substitution pass skips it, the reference survives into the bundle, and it evaluates against an empty `process.env` in the browser: every value `undefined`, and on our `firebaseWebRest.ts` that means the REST `key=` param is empty and sign-in fails, **no error naming the cause**. Invisible to every test (Jest has a real `process.env`, so the helper works there) and to typecheck — it fails only in an actual web export, which is exactly the S0.2 `getTokens()` shape: green everywhere except the one environment that ships. **Rule: every `EXPO_PUBLIC_*` read must be a direct static member access.** Verify an export actually baked a value with `Select-String` for the literal in `dist/_expo/static/js/web/*.js` before trusting it. Also: Expo caches aggressively — use `expo export --clear` when a value ought to have changed and the bundle still shows the old one. Hit at S0.4 on the first web preview export.
- **The release build needs a ninja ≥ 1.12 on Windows; the SDK's bundled 1.10 cannot build it, and every debug build works regardless.** `assembleRelease` dies with `ninja: error: Stat(…RNGestureHandlerDetectorShadowNode.cpp.o): Filename longer than 260 characters` — React Native's codegen mirrors each module's *source path* into its object-file path, so the generated names pass 260 characters before anything of ours contributes. **Two traps in one.** (1) *Windows is not the limiter*: `LongPathsEnabled` is already 1 here, so the advice every search result gives ("enable long paths") does nothing — ninja only learned long paths in **1.12** (`longPathAware` manifest + Unicode Win32 APIs), and Android SDK CMake 3.22.1 bundles **1.10.2**. (2) *Only release trips it* — the debug variant's paths sit just under the line, so `npm run android` works forever while the artifact that actually ships cannot be built; the toolchain looks healthy right up to the moment it has to produce a release. **Fixed and should stay fixed:** `winget install Ninja-build.Ninja` once per machine, and `mobile/plugins/withLongPathNinja.js` injects `-DCMAKE_MAKE_PROGRAM=<that ninja>` into `defaultConfig.externalNativeBuild.cmake.arguments` at every prebuild (override the ninja lookup with `LARGATA_NINJA`; the plugin warns loudly rather than failing if none ≥ 1.12 is found, since debug work is unaffected). **The mechanism matters and a wrong turn is recorded in the plugin:** there is no `android.ninjaPath` gradle property — setting one is accepted and does nothing, failing with the identical error, because *CMake*, not AGP, chooses ninja. Never "fix" it by editing `android/gradle.properties` or `android/app/build.gradle` — both generated and gitignored, so the edit vanishes at the next prebuild. Hit at S0.4, on the first `assembleRelease` this repo ever ran.
- **`android/` and `ios/` are generated, not committed (CNG).** `expo prebuild` regenerates them from `app.json` + config plugins; anything hand-edited inside them is erased without warning. Native customization goes through a **config plugin** (`mobile/plugins/`), which is also why a fresh clone must run `npx expo prebuild` before it can build.
- **A fresh clone cannot prebuild without `mobile/google-services.json`** — `app.json` names it (`android.googleServicesFile`), the RNFirebase plugin hard-fails without it, and it is gitignored by design (it holds an API key; grilling Q4). Download it from the Firebase console → `largata-dev` → Project settings → the `com.largata.app` Android app. The failure is an error about a *config field*, which does not obviously mean "download a file". CI is unaffected: the mobile job never prebuilds.
- **Google sign-in: pass BOTH tokens to `GoogleAuthProvider.credential(idToken, accessToken)`, via `GoogleSignin.getTokens()`.** RNFirebase v25's *JS* layer says an idToken alone is fine (it only rejects when both are null) but its *native* layer then throws `[auth/unknown] Exception in HostFunction: accessToken cannot be empty` — **the two layers disagree, so the TypeScript signature lies**. `signIn()` returns only the idToken; `getTokens()` returns both. Hit at S0.2 and invisible to every unit test: the mock returned an idToken and the JS layer accepted it, so the failure existed only on a device. The Jest mock now enforces the *native* contract, not the JS one.
- **To reach a screen the UI has no button for, deep-link it: `adb shell am start -a android.intent.action.VIEW -d "largata://<route>"`.** The scheme is **`largata`** (`app.json` → `scheme`), *not* the applicationId `com.largata.app` — guessing the latter yields "unable to resolve Intent", which reads as "deep-linking isn't wired" when it is. This runs against the app's **live session**, so it exercises any route as the signed-in traveler with no debugger, no Metro console, and — the point — no digging credentials out of app-private storage. It is how S0.3's guard AC was closed: traveler B could not reach traveler A's itinerary by tapping (correctly, it is not in B's list), but `largata://itineraries/<A's id>` asked for it directly and got the 404.
- Share-sheet capture requires an Expo **dev build** — plain Expo Go cannot load the native share extension (ADR-004).
- Local **iOS builds require macOS + Xcode** — Android builds run anywhere via Gradle. **The recorded answer is EAS at the iOS activation** (or earlier, if Android release cadence makes manual builds a cost) — no Mac purchase planned (ADR-010).
- **`expo install --check` reports `react@19.2.7 - expected version: 19.2.3` — this is expected; do not "fix" it.** Expo SDK 57 pins `react@19.2.3`, but `expo-router` pulls `react-dom@19.2.7`, which requires `react@^19.2.7`; the two Expo-published constraints contradict each other. We hold `react@19.2.7` (the value that makes `npm install` resolve without `--legacy-peer-deps`). Verified at S0.1: typecheck passes, Jest passes, and `expo export` bundles for Android. Re-check when the SDK bumps — if Expo aligns its own pins, drop back to the SDK's version.
  - **Corollary: `react` and `react-dom` must be pinned to the same exact version, and `npx expo install react-dom` breaks that** — it installs the SDK's `19.2.3` against our `react@19.2.7`, and the web bundle then dies at runtime with *"Incompatible React versions"* on a white screen. After any `expo install` that touches React, check both versions match. (Hit at S0.1 when adding web support.)
- **`Alert.alert` is a NO-OP on react-native-web — every RN API that fronts a native dialog needs a web fork, and the failure is total silence.** react-native-web's `Alert` is literally `class Alert { static alert() {} }`. Any control whose whole job is to say something — a confirm, a "coming soon", an error the traveler must see — therefore does **nothing at all** in the browser while working perfectly on the device. S1.3 shipped its entire grey-out shell (four nav tabs, two media tiles, the Preview CTA) as dead clicks on the web this way, violating the AC that *defined* the ticket. **Invisible to everything:** the unit test asserted the `Alert` call and passed (Jest resolves the `.native` fork), the ITs never touch the client, and a preview driver that only checks screens *render* sees nothing wrong — "renders on web" is not "works on web". Found by a human tapping the preview. **Fix:** platform-fork the helper (`x.native.ts` / `x.web.ts`, the auth and date-picker pattern), keep the shared wording in a third module so it cannot drift, and verify by *driving* the container — intercept `window.alert` in CDP, since headless Chrome swallows native dialogs. The same suspicion applies to any other RN API whose web implementation is a stub. Hit at S1.3.
- **Expo Go is abandoned — resolved at S0.2, do not go back.** Two independent blockers, not one: (1) the store client lagged SDK 57, so phones reported *"Project is incompatible with this version of Expo Go"*; (2) even the CDN-sideloaded SDK 57 client on the emulator (which bypasses the store entirely, so the lag theory was wrong) **segfaults natively** — `Fatal signal 11 (SIGSEGV), tid mqt_v_js` — after our bundle loads. Nothing our code caused or can fix. The dev-build (`npm run android` → `expo run:android`) compiles our own SDK into our own container and makes both irrelevant permanently.
- **Testcontainers 2.x renamed everything** — artifacts (`postgresql` → `testcontainers-postgresql`) *and* packages (`org.testcontainers.containers` → `org.testcontainers.postgresql`; the old package is a deprecated shim, and the container class is no longer generic). The 1.x coordinates 404 at 2.x, which reads like "2.x is broken" — it isn't. Versions come from the Boot BOM; never hand-pin them.
- **Boot 4 moved Flyway's auto-configuration out of the JPA starter** into `spring-boot-starter-flyway`. With only `flyway-core` on the classpath, Flyway silently never runs — the app boots green and no migration applies. Use the starter.
- **Boot 4.1 ships Jackson 3: `ObjectMapper` is `tools.jackson.databind.ObjectMapper`, not `com.fasterxml.jackson.databind`.** Only `jackson-annotations` still lives under `com.fasterxml`. Every Jackson-2-era example (i.e. the entire internet's Spring corpus) fails to resolve; write methods also no longer declare `IOException`. Hit at S0.2 in the auth entry point.
- **Boot 4 split autoconfiguration into per-technology modules, and the packages moved with them** — e.g. `OAuth2ResourceServerProperties` is now in `org.springframework.boot.security.oauth2.server.resource.autoconfigure`, not `org.springframework.boot.autoconfigure.security.oauth2.resource`. When an autoconfigure import won't resolve, the class almost certainly still exists under a new path: find it (`jar tf` the module) rather than assuming it's gone. Same family as the Flyway-starter gotcha above.
- **An unordered `@Component` filter runs *behind* Spring Security (chain order -100), not in front of it.** Boot gives unordered filters `LOWEST_PRECEDENCE`, so anything a rejected request needs — a `traceId` in the MDC, say — is never set, and the 401 renders without it. Invisible on the happy path (a 200 gets it either way). Any filter that must cover auth failures needs an explicit `@Order(HIGHEST_PRECEDENCE)`. Conversely, a filter that needs the *principal* must be registered inside the chain (`addFilterAfter(...)`) — as a `@Component` it runs too early and silently does nothing. Both hit at S0.2.
- **Spring Security's resource-server DSL installs its own `AuthenticationEntryPoint`**, so setting one on `exceptionHandling()` alone only covers "no credentials at all" — a *bad* token (expired/forged/malformed) answers through `BearerTokenAuthenticationEntryPoint` instead, with an empty body. One situation, two response shapes. Set it on `oauth2ResourceServer(...)` **and** `exceptionHandling(...)`. Hit at S0.2; caught only because the AC tested four token flavors rather than one.
- **`@Transactional` on a method called from within the same bean does nothing** (self-invocation bypasses the proxy). And a constraint violation marks its transaction rollback-only, so "catch the violation, then re-read the winner's row" fails *in the same transaction* — the recovery read dies too. Insert-on-conflict recovery therefore needs the insert in a **separate bean** with `REQUIRES_NEW` (and `saveAndFlush`, so the violation is raised inside the try, not deferred to commit). Hit at S0.2 in Traveler provisioning.
- **Testcontainers hangs forever at `Testcontainers version: 2.0.5` — and Docker is not the problem.** No container starts, nothing times out, while `docker ps`, `docker version` and even `docker run postgres:17-alpine` all answer in milliseconds. That contradiction is the tell: **`jstack` the surefire fork** (find its pid via `Get-CimInstance Win32_Process -Filter "Name='java.exe'"`, look for `surefire` in the CommandLine) and the `main` thread is parked in `docker-java`'s `HijackingHttpRequestExecutor`, waiting on a response header the daemon never sends. Ryuk — Testcontainers' reaper — opens a *hijacked* HTTP connection over the Windows named pipe and that path stalls; the CLI never hijacks, which is why it looks healthy. Workaround: `TESTCONTAINERS_RYUK_DISABLED=true` (Ryuk only reaps leftover containers; without it, clean them by hand if a run dies). **Kill only your own JVMs, by pid** — never `taskkill /IM java.exe`, which also takes out the IDE's language server, SonarLint and Android Studio. Hit repeatedly at S0.3; intermittent (the same suite passed once at 311s before wedging on every later attempt).
- **A *second*, different Testcontainers wedge on Windows: the Docker credential helper hangs — and the image already being present locally is what makes it baffling.** Symptom identical to the Ryuk one (no container in `docker ps`, no timeout, the fork alive), but `jstack`ing the surefire fork shows `main` parked in **`RegistryAuthLocator.runCredentialProgram → ProcessExecutor.waitFor`**, not `HijackingHttpRequestExecutor`. Before using even a *locally cached* image, Testcontainers resolves registry auth by shelling out to the credential helper named in `~/.docker/config.json` (`"credsStore": "desktop"` → `docker-credential-desktop.exe`), and that subprocess never returns here — so `postgres:18-alpine` is sitting in `docker images` while the run hangs on an auth lookup it does not even need. `TESTCONTAINERS_RYUK_DISABLED=true` does **not** fix this one (different code path). **Workaround: point `DOCKER_CONFIG` at a dir holding a minimal `{}` `config.json`** (no `credsStore`), so Testcontainers finds no credential program to run: `mkdir cfg && echo '{}' > cfg/config.json && DOCKER_CONFIG=$(pwd)/cfg mvn …`. Then the container starts in seconds and the migration/tests proceed. Hit at S1.2; the two wedges can masquerade as each other, so **`jstack` and read the actual parked frame before picking a fix** — Ryuk-disable vs. DOCKER_CONFIG are not interchangeable.
- **A query against a deployed rung must name the database it belongs to — "no rows" from the wrong one is indistinguishable from "the answer is no".** Deployed dev's backend connects to **`postgres.railway.internal:5432/railway`** — database **`railway`**, a name nothing in this repo would let you guess (it is Railway's default, not ours). At S1.1 the post-deploy check was handed over as bare SQL, returned zero rows, and was read by agent *and* owner as *"the migrations never ran"* — sending the investigation to CI and `gh auth` while the deploy log showed both migrations had applied cleanly minutes earlier. The null result was ambiguous and nobody noticed. **Name the database (and the environment) in any query whose answer you intend to act on**; same family as ADR-006's environment identity and S0.6's phantom Cloud project — *the thing you are talking to must be identified, never assumed*.
- **A check whose two outcomes are indistinguishable is not a check — and this repo has now been burned by one three times.** S1.1's version: to detect whether a deploy had landed, the agent created an itinerary on deployed dev and viewed it, reasoning only the new build could serve the view. Both builds return 200 (old: `owner_id`; new: a membership row) — same response, opposite mechanisms. It happened to hit the new build, so it gave the *right answer by an invalid method*, which is worse than a wrong one: it would have "confirmed" the deploy just as happily before it happened. Precedents: S0.6's `/gsi/button` watcher (400 forever while the button worked) and its `curl`-vs-browser cousin. **Before trusting a probe, state what its failure looks like.** If you cannot, it has no failure mode and proves nothing. The signals that *were* discriminating here — `flyway_schema_history` and the deploy log — were both outside the API.
- **A data migration is invisible to every test surface this repo owns — it runs against zero rows and "passes" as a no-op.** The local stack is fresh-DB-every-redeploy, Testcontainers boots empty schemas, CI likewise: on all of them a backfill does nothing and reports success, so the suite is green whether the SQL is right, subtly wrong, or a typo. The only database holding the rows it targets is the deployed rung. **The fix is a migration-stepping IT** (`WorkspaceBackfillIT` is the pattern): its *own* container — never `PostgresTestBase`'s singleton, which is shared by the run and already migrated to head before the class loads — Flyway `.target(V(n-1)).migrate()`, seed the legacy shape via **raw SQL** (going through the service would create the very rows the fixture must lack), then `.target(V(n)).migrate()` and assert. **And check the test can fail:** S1.1 sabotaged V5's timestamps to `now()` and confirmed the test caught it with the right diagnosis. A migration test that passes against a no-op is worse than none. Pair it with a post-merge check on the rung that has the data — the shape rehearses locally by planting a legacy row in the running stack and applying the migration file itself through psql (S1.1 did: owner 404 → apply → 200). Same family as S0.2's `getTokens()`: green everywhere except the one environment that ships. Hit at S1.1, on the first data migration this repo ever wrote.
- **`@Enumerated(STRING)` writes the enum's NAME (`OWNER`), and any SQL that tests that value is a silent contract with Hibernate.** V3's `state TEXT DEFAULT 'draft'` is dead weight — Hibernate overrides it with `DRAFT` on every write — and that lower-case spelling is a trap for the next migration that copies it. S1.1's first draft of V4 wrote `CREATE UNIQUE INDEX … WHERE role = 'owner'` to enforce INV-4: the predicate would have matched **zero rows**, so the index would create successfully, cost nothing, and enforce **nothing** — the invariant unguarded, no error, ever, and no other test would have noticed (a partial index on a never-matching predicate is the quietest possible failure). Caught before it landed; `MembershipStorageIT` now pins the spelling. **The distinction to keep:** an enum's *storage* spelling is a contract between Hibernate and SQL; the API's spelling (`wireName()`) is a *different* contract that happens to look similar. Any constraint, index or `WHERE` naming an enum value needs a test that would fail if the spelling moved.
- **A Spring `@MockitoSpyBean` wraps the `@Transactional` proxy, so `when(spy).method(…)` runs the transaction interceptor during *stub setup*.** With `Propagation.MANDATORY` on the method, stubbing throws `IllegalTransactionStateException` before the test body starts — surfacing as a Mockito `InvalidUseOfMatchersException` about misplaced matchers, which names nothing about the real cause and sends you hunting in the test's syntax. Stub one layer lower (the repository the service saves through) — no AOP proxy in the way, and the failure is injected where a real one would happen. Hit at S1.1.
- **Testcontainers: use the singleton-container pattern (`PostgresTestBase`), never `@Testcontainers`/`@Container` on a shared base.** Those annotations manage the container *per test class*; Spring caches contexts *by configuration*. The moment a second context exists (e.g. a test with a different `@ActiveProfiles`), a cached context keeps a datasource pointing at a container JUnit already stopped. **Signature: 30-second JDBC timeouts and 503s in test classes you did not touch**, right after adding a test with a different profile. Hit for real at S0.1 when the CORS profile tests landed.
- **Failsafe matches `*IT` on the *outer* class only** — integration tests nested as `static` inner classes are silently skipped, which looks exactly like passing. One test class per file. (Also hit at S0.1: a CORS test "passed" without ever running.)
