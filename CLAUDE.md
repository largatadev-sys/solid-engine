## State â€” always maintain

BUILD_STATUS â†’ `BUILD_STATUS.md`. Every story that lands updates its row â€” **status + spec link, and nothing else**: no SHAs, no branch names, no summary of what the story proved. The tickets carry the detail and git answers "where does it live"; anything duplicated in the tracker is a fact that rots. **Update the row before the merge, in the last commit on the feature branch** â€” updating after means committing straight to `dev`, which this workflow doesn't allow. Every change that wasn't a planned story gets a line in the off-epic ledger. Keeping BUILD_STATUS current is **non-negotiable** â€” a stale tracker misleads the next session with authority, which is worse than no tracker. Anything raised along the way (idea, issue, deferred feature) is captured in the epic map's backlog â€” that is its one home. Regression checklist â†’ `REGRESSION_CHECKLIST.md` (a bug that escapes to a human adds a line; a recurring line graduates into the Maestro smoke suite).

## Hard rules (this codebase)

- **All workspace access goes through the authorization guard; never query workspace tables directly.** Workspace-scoped service methods take the resolved `Membership` as a parameter (Artifact 03). No inline authority checks anywhere.
- **All capability gating goes through the entitlement service** â€” `can(traveler, capability)`; never inline tier checks (ADR-009). v1: it returns full access.
- **The ledger is append-only.** Corrections, waivers, settlements, reassignments are new Transfer entries â€” never UPDATE or DELETE on ledger rows (INV-8). Splits must sum to the expense total transactionally (INV-7).
- **API changes are additive only within /v1** â€” never rename, retype, remove, or change semantics of anything shipped (ADR-008; old app versions live for weeks).
- **Modules reference each other by ID + service interface only** â€” never another module's tables or internals (ADR-002).
- **Mobile: no raw fetch/API calls in UI code** â€” everything through the repository/local-cache layer's typed apiClient (ADR-001, P6).
- **Never log secrets, tokens, passwords, raw payloads, or PII.** Reference entities by ID (P3).

## Never commit secrets

No `.env` or any file containing credentials, keys, tokens, passwords, or **PII** enters a commit â€” ever. This is P3 (never-log-secrets) extended from logs to commits: secrets never enter *any* durable artifact. Two layers, no third-party tool required:

- **Structural guarantee (the real defence): `.gitignore` the secret files before the first commit.** `.env`, `.env.local`, anything holding credentials â€” gitignored so they *cannot* be staged. Real secrets live only in gitignored files locally and in the platform's env-var UI in prod; the repo carries `.env.example` with placeholders only. If nothing sensitive is ever tracked, there is nothing to leak â€” it does not depend on anyone remembering.
- **Backstop (soft): scan the staged diff before every commit.** Check the staged content for secret/PII patterns (`API_KEY=`, `SECRET`, `PASSWORD=`, tokens, long high-entropy strings, any `.env` in the staged files) and refuse to commit on a match. It is a backstop *because* it's an instruction â€” lead with the gitignore, never let the scan become an excuse to relax it.

A committed secret is compromised even if a later commit deletes it (it stays in history) â€” **rotate it.**

## Git workflow (promotion pipeline)

Branchâ†’environment mapping: `main` â†’ **production** (protected base, real data) Â· `preprod` â†’ a **pristine copy of production** (the verify gate; data mirrors prod) Â· `dev` â†’ **long-lived shared preview** (persistent; reseed at the developer's discretion, *not* ephemeral) Â· **local** â†’ the **ephemeral full-stack instance** (fresh DB every redeploy) where feature branches are built and tested.

Flow: feature branches (`feature/<story-id>-<slug>`) are tested on the **local ephemeral instance**, then **squash-merged into `dev`**. Promotion: **cherry-pick** the verified changes from `dev` â†’ `preprod`, verify against the prod-copy, then promote `preprod` â†’ `main`. Commit convention: `feat(scope): <story-id> â€¦` / `fix(scope): <story-id> â€¦` â€” **the story id in the message is mandatory**: it is how commits are located (`git log --grep <story-id>`), since BUILD_STATUS stores no SHAs. *Commit messages carry **no agent signature** â€” no `Co-Authored-By`, no "Generated with" trailer, no attribution line; just the conventional message.*

**Promotions are propose-first.** The agent may run routine git directly â€” create the feature branch, commit to it â€” but it **proposes and waits for approval before any promotion**: squash-merging `feature â†’ dev`, cherry-picking `dev â†’ preprod`, or promoting `preprod â†’ main`. Promotions cross environments and are hard to reverse; each is a checkpoint, not an autonomous step.

**Footgun to watch:** cherry-picking after squash-merges can pull a change whose dependency you didn't pick â€” a subset that built in `dev` may not build in `preprod`. Track inter-change dependencies before promoting. **The mobile train is separate** (Artifact 04, ADR-010 â€” Android-only until the iOS activation): emulator/dev-build (local, non-EAS) â†’ local backend Â· Play internal testing â†’ preprod Â· Play release â†’ prod. Release builds are made locally (Expo prebuild + Gradle) and uploaded manually.

## Gotchas (grows during the build)

- **The two release trains need different JDKs on one machine: backend = Java 25, Android = Java 21.** `JAVA_HOME` points at JDK 25 (S0.1's backend requirement), and the Android Gradle Plugin does not support it â€” the build dies at `configureCMakeDebug` reporting only *"WARNING: A restricted method in java.lang.System has been called"*, a JDK warning masquerading as the whole error, which reads like a broken toolchain rather than a version mismatch. **Already fixed and should stay fixed:** `mobile/plugins/withAndroidJdk.js` pins `org.gradle.java.home` to Android Studio's bundled JBR at every prebuild (override with `LARGATA_ANDROID_JAVA_HOME`). Never "fix" this by editing `android/gradle.properties` â€” that directory is generated and gitignored, so the edit vanishes at the next prebuild and the inscrutable CMake error returns. Hit at S0.2.
- **The Firebase JS SDK white-screens the web export until `metro.config.js` disables `unstable_enablePackageExports`.** Symptom: `Uncaught Error: Component auth has not been registered yet` at module load, so React never renders and the page is blank â€” with a clean network tab (the bundle 200s), which sends you hunting in the wrong place. Cause: Expo SDK 53+ defaults Metro's package-`exports` resolver on, and `@firebase/auth` exports a **`react-native`** condition (`dist/rn/index.js`) distinct from `browser`/`default`. Metro is the RN bundler, so it applies `react-native` even for `expo export --platform web`, pulling Firebase's RN auth build into the *browser* bundle â€” its component registration never runs there, and `getAuth`/`initializeAuth` throw. **Fix that actually worked on our versions (firebase 12, Expo SDK 57): `metro.config.js` steering the resolver *conditions* â€” `resolver.unstable_conditionNames = ['browser','require','import']` (+ `unstable_enablePackageExports = true`, `sourceExts.push('cjs')`).** The widely-cited community fix (`unstable_enablePackageExports = false`, expo/expo#36588) **did not move it here** â€” the RN condition still won and the RN build stayed in the web bundle. **Verify at the bundle, not in the browser:** grep `dist/_expo/static/js/web/*.js` for `getReactNativePersistence` â€” PRESENT means the RN build is still bundled (broken), GONE means the browser build resolved (fixed). That check is the whole debugging loop; reloading the page to see if it renders is the slow way and cost several wasted cycles at S0.4. This steers `firebase/*` resolution for *both* platform targets, but native imports `@react-native-firebase`, not `firebase/*`, so native is untouched â€” still, **re-verify the native bundle** (`expo export --platform android`, it stayed green at S0.4). Two independent traps stack here: even with correct resolution, read `EXPO_PUBLIC_*` by direct static access (below) and use `initializeAuth(app, {persistence: browserLocalPersistence})` not `getAuth` (explicit registration + sessions survive refresh). All three hit at S0.4's first browser load; none is visible to Jest or typecheck â€” the S0.2 "green everywhere except the environment that ships" pattern, web edition.
- **Expo inlines `EXPO_PUBLIC_*` on web by literal text substitution of `process.env.EXPO_PUBLIC_X` â€” a computed `process.env[name]` is invisible to it and silently ships unresolved.** There is no runtime environment in a web bundle; Expo bakes these vars at export time by find-and-replacing the *exact source text* `process.env.EXPO_PUBLIC_FOO`. Read one indirectly â€” `const v = process.env[name]`, a `required(name)` helper, destructuring, anything non-literal â€” and the substitution pass skips it, the reference survives into the bundle, and it evaluates against an empty `process.env` in the browser: every value `undefined`, and on our `firebaseWebApp.ts` that means Firebase init with no config, sign-in dead, **no error naming the cause**. Invisible to every test (Jest has a real `process.env`, so the helper works there) and to typecheck â€” it fails only in an actual web export, which is exactly the S0.2 `getTokens()` shape: green everywhere except the one environment that ships. **Rule: every `EXPO_PUBLIC_*` read must be a direct static member access.** Verify an export actually baked a value with `Select-String` for the literal in `dist/_expo/static/js/web/*.js` before trusting it. Also: Expo caches aggressively â€” use `expo export --clear` when a value ought to have changed and the bundle still shows the old one. Hit at S0.4 on the first web preview export.
- **The release build needs a ninja â‰Ą 1.12 on Windows; the SDK's bundled 1.10 cannot build it, and every debug build works regardless.** `assembleRelease` dies with `ninja: error: Stat(â€¦RNGestureHandlerDetectorShadowNode.cpp.o): Filename longer than 260 characters` â€” React Native's codegen mirrors each module's *source path* into its object-file path, so the generated names pass 260 characters before anything of ours contributes. **Two traps in one.** (1) *Windows is not the limiter*: `LongPathsEnabled` is already 1 here, so the advice every search result gives ("enable long paths") does nothing â€” ninja only learned long paths in **1.12** (`longPathAware` manifest + Unicode Win32 APIs), and Android SDK CMake 3.22.1 bundles **1.10.2**. (2) *Only release trips it* â€” the debug variant's paths sit just under the line, so `npm run android` works forever while the artifact that actually ships cannot be built; the toolchain looks healthy right up to the moment it has to produce a release. **Fixed and should stay fixed:** `winget install Ninja-build.Ninja` once per machine, and `mobile/plugins/withLongPathNinja.js` injects `-DCMAKE_MAKE_PROGRAM=<that ninja>` into `defaultConfig.externalNativeBuild.cmake.arguments` at every prebuild (override the ninja lookup with `LARGATA_NINJA`; the plugin warns loudly rather than failing if none â‰Ą 1.12 is found, since debug work is unaffected). **The mechanism matters and a wrong turn is recorded in the plugin:** there is no `android.ninjaPath` gradle property â€” setting one is accepted and does nothing, failing with the identical error, because *CMake*, not AGP, chooses ninja. Never "fix" it by editing `android/gradle.properties` or `android/app/build.gradle` â€” both generated and gitignored, so the edit vanishes at the next prebuild. Hit at S0.4, on the first `assembleRelease` this repo ever ran.
- **`android/` and `ios/` are generated, not committed (CNG).** `expo prebuild` regenerates them from `app.json` + config plugins; anything hand-edited inside them is erased without warning. Native customization goes through a **config plugin** (`mobile/plugins/`), which is also why a fresh clone must run `npx expo prebuild` before it can build.
- **A fresh clone cannot prebuild without `mobile/google-services.json`** â€” `app.json` names it (`android.googleServicesFile`), the RNFirebase plugin hard-fails without it, and it is gitignored by design (it holds an API key; grilling Q4). Download it from the Firebase console â†’ `largata-dev` â†’ Project settings â†’ the `com.largata.app` Android app. The failure is an error about a *config field*, which does not obviously mean "download a file". CI is unaffected: the mobile job never prebuilds.
- **Google sign-in: pass BOTH tokens to `GoogleAuthProvider.credential(idToken, accessToken)`, via `GoogleSignin.getTokens()`.** RNFirebase v25's *JS* layer says an idToken alone is fine (it only rejects when both are null) but its *native* layer then throws `[auth/unknown] Exception in HostFunction: accessToken cannot be empty` â€” **the two layers disagree, so the TypeScript signature lies**. `signIn()` returns only the idToken; `getTokens()` returns both. Hit at S0.2 and invisible to every unit test: the mock returned an idToken and the JS layer accepted it, so the failure existed only on a device. The Jest mock now enforces the *native* contract, not the JS one.
- **To reach a screen the UI has no button for, deep-link it: `adb shell am start -a android.intent.action.VIEW -d "largata://<route>"`.** The scheme is **`largata`** (`app.json` â†’ `scheme`), *not* the applicationId `com.largata.app` â€” guessing the latter yields "unable to resolve Intent", which reads as "deep-linking isn't wired" when it is. This runs against the app's **live session**, so it exercises any route as the signed-in traveler with no debugger, no Metro console, and â€” the point â€” no digging credentials out of app-private storage. It is how S0.3's guard AC was closed: traveler B could not reach traveler A's itinerary by tapping (correctly, it is not in B's list), but `largata://itineraries/<A's id>` asked for it directly and got the 404.
- Share-sheet capture requires an Expo **dev build** â€” plain Expo Go cannot load the native share extension (ADR-004).
- Local **iOS builds require macOS + Xcode** â€” Android builds run anywhere via Gradle. **The recorded answer is EAS at the iOS activation** (or earlier, if Android release cadence makes manual builds a cost) â€” no Mac purchase planned (ADR-010).
- **`expo install --check` reports `react@19.2.7 - expected version: 19.2.3` â€” this is expected; do not "fix" it.** Expo SDK 57 pins `react@19.2.3`, but `expo-router` pulls `react-dom@19.2.7`, which requires `react@^19.2.7`; the two Expo-published constraints contradict each other. We hold `react@19.2.7` (the value that makes `npm install` resolve without `--legacy-peer-deps`). Verified at S0.1: typecheck passes, Jest passes, and `expo export` bundles for Android. Re-check when the SDK bumps â€” if Expo aligns its own pins, drop back to the SDK's version.
  - **Corollary: `react` and `react-dom` must be pinned to the same exact version, and `npx expo install react-dom` breaks that** â€” it installs the SDK's `19.2.3` against our `react@19.2.7`, and the web bundle then dies at runtime with *"Incompatible React versions"* on a white screen. After any `expo install` that touches React, check both versions match. (Hit at S0.1 when adding web support.)
- **Expo Go is abandoned â€” resolved at S0.2, do not go back.** Two independent blockers, not one: (1) the store client lagged SDK 57, so phones reported *"Project is incompatible with this version of Expo Go"*; (2) even the CDN-sideloaded SDK 57 client on the emulator (which bypasses the store entirely, so the lag theory was wrong) **segfaults natively** â€” `Fatal signal 11 (SIGSEGV), tid mqt_v_js` â€” after our bundle loads. Nothing our code caused or can fix. The dev-build (`npm run android` â†’ `expo run:android`) compiles our own SDK into our own container and makes both irrelevant permanently.
- **Testcontainers 2.x renamed everything** â€” artifacts (`postgresql` â†’ `testcontainers-postgresql`) *and* packages (`org.testcontainers.containers` â†’ `org.testcontainers.postgresql`; the old package is a deprecated shim, and the container class is no longer generic). The 1.x coordinates 404 at 2.x, which reads like "2.x is broken" â€” it isn't. Versions come from the Boot BOM; never hand-pin them.
- **Boot 4 moved Flyway's auto-configuration out of the JPA starter** into `spring-boot-starter-flyway`. With only `flyway-core` on the classpath, Flyway silently never runs â€” the app boots green and no migration applies. Use the starter.
- **Boot 4.1 ships Jackson 3: `ObjectMapper` is `tools.jackson.databind.ObjectMapper`, not `com.fasterxml.jackson.databind`.** Only `jackson-annotations` still lives under `com.fasterxml`. Every Jackson-2-era example (i.e. the entire internet's Spring corpus) fails to resolve; write methods also no longer declare `IOException`. Hit at S0.2 in the auth entry point.
- **Boot 4 split autoconfiguration into per-technology modules, and the packages moved with them** â€” e.g. `OAuth2ResourceServerProperties` is now in `org.springframework.boot.security.oauth2.server.resource.autoconfigure`, not `org.springframework.boot.autoconfigure.security.oauth2.resource`. When an autoconfigure import won't resolve, the class almost certainly still exists under a new path: find it (`jar tf` the module) rather than assuming it's gone. Same family as the Flyway-starter gotcha above.
- **An unordered `@Component` filter runs *behind* Spring Security (chain order -100), not in front of it.** Boot gives unordered filters `LOWEST_PRECEDENCE`, so anything a rejected request needs â€” a `traceId` in the MDC, say â€” is never set, and the 401 renders without it. Invisible on the happy path (a 200 gets it either way). Any filter that must cover auth failures needs an explicit `@Order(HIGHEST_PRECEDENCE)`. Conversely, a filter that needs the *principal* must be registered inside the chain (`addFilterAfter(...)`) â€” as a `@Component` it runs too early and silently does nothing. Both hit at S0.2.
- **Spring Security's resource-server DSL installs its own `AuthenticationEntryPoint`**, so setting one on `exceptionHandling()` alone only covers "no credentials at all" â€” a *bad* token (expired/forged/malformed) answers through `BearerTokenAuthenticationEntryPoint` instead, with an empty body. One situation, two response shapes. Set it on `oauth2ResourceServer(...)` **and** `exceptionHandling(...)`. Hit at S0.2; caught only because the AC tested four token flavors rather than one.
- **`@Transactional` on a method called from within the same bean does nothing** (self-invocation bypasses the proxy). And a constraint violation marks its transaction rollback-only, so "catch the violation, then re-read the winner's row" fails *in the same transaction* â€” the recovery read dies too. Insert-on-conflict recovery therefore needs the insert in a **separate bean** with `REQUIRES_NEW` (and `saveAndFlush`, so the violation is raised inside the try, not deferred to commit). Hit at S0.2 in Traveler provisioning.
- **Testcontainers hangs forever at `Testcontainers version: 2.0.5` â€” and Docker is not the problem.** No container starts, nothing times out, while `docker ps`, `docker version` and even `docker run postgres:17-alpine` all answer in milliseconds. That contradiction is the tell: **`jstack` the surefire fork** (find its pid via `Get-CimInstance Win32_Process -Filter "Name='java.exe'"`, look for `surefire` in the CommandLine) and the `main` thread is parked in `docker-java`'s `HijackingHttpRequestExecutor`, waiting on a response header the daemon never sends. Ryuk â€” Testcontainers' reaper â€” opens a *hijacked* HTTP connection over the Windows named pipe and that path stalls; the CLI never hijacks, which is why it looks healthy. Workaround: `TESTCONTAINERS_RYUK_DISABLED=true` (Ryuk only reaps leftover containers; without it, clean them by hand if a run dies). **Kill only your own JVMs, by pid** â€” never `taskkill /IM java.exe`, which also takes out the IDE's language server, SonarLint and Android Studio. Hit repeatedly at S0.3; intermittent (the same suite passed once at 311s before wedging on every later attempt).
- **Testcontainers: use the singleton-container pattern (`PostgresTestBase`), never `@Testcontainers`/`@Container` on a shared base.** Those annotations manage the container *per test class*; Spring caches contexts *by configuration*. The moment a second context exists (e.g. a test with a different `@ActiveProfiles`), a cached context keeps a datasource pointing at a container JUnit already stopped. **Signature: 30-second JDBC timeouts and 503s in test classes you did not touch**, right after adding a test with a different profile. Hit for real at S0.1 when the CORS profile tests landed.
- **Failsafe matches `*IT` on the *outer* class only** â€” integration tests nested as `static` inner classes are silently skipped, which looks exactly like passing. One test class per file. (Also hit at S0.1: a CORS test "passed" without ever running.)
