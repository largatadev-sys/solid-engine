# 07 · Epic Map — Largata  `[MVP-THIN — launch epics named; near-term stories detailed]`

**Architect's question:** *What is the whole territory, and what is the smallest slice of it that tests the idea?*

**The epic map is durable — the living backlog, not a design-time snapshot.** Anything raised over the life of the product — an improvement, an issue, a deferred idea, a new feature — has one home: the backlog here. Raising something *means* capturing it in this map; it waits as a placeholder and is elaborated into stories only when picked up. Created during design, **maintained forever after** — it does not stop being the backlog when the MVP ships.

_Status: **proposed — pending founder ratification.** Launch scope = Epics 0–6 (founder decision: the ledger is **in** launch — the hypothesis is "plan + costs + record"; costs stay). Stories are elaborated **just-in-time**: Epic 0's are agent-ready now; later epics carry story lists at slice level, elaborated when pulled. Only the next few stories are ever in flux — by design._

---

## Launch sequence & dependencies

```
E0 skeleton → E1 collaborative planning → E2 decisions → E3 the record → E4 social surface → E5 ledger → E6 unfurler
```

- **E0 is the walking skeleton = the MVP's spine:** one thin path through every layer, deployed to the prod target and installable via TestFlight/internal. It validates both release trains, the guard, and the pipeline while the cost of being wrong is one feature.
- **E6 last is deliberate:** items function as **bare links + manual fields from E1 onward** (the unfurler's own degradation mode is the baseline), so nothing upstream waits on it. Its real risk (bot-blocking/coverage) is de-risked by the **register #8 spike, run early and independently** — an afternoon script, no story dependency. If the spike surprises badly, ADR-007's buy-fallback absorbs it without moving the epic.
- **E5 (ledger) is in launch scope** — the biggest cuttable if runway tightens, but cut it and the hypothesis loses "costs."
- **The validation gate sits after E6 ships to alpha** — criteria per register #1 (COO, signed before alpha).

- **A cold visit fires `GET /v1/itineraries` before auth resolves and takes a 401** — *(raised at S0.5, 2026-07-17, during the post-merge check on `founders.largata.com`; **pre-existing, not an S0.5 regression** — S0.5 touched no query/hook/routing code)* with empty `localStorage` and no session, the trips query fires anyway, the backend correctly 401s (the guard working), and the auth gate then routes to sign-in. **Harmless today**: nothing leaks, nothing is user-visible, the screen is right. But it is a wasted unauthenticated round-trip on every cold visit, and it is the kind of log line that reads as a bug to whoever next reads the access logs. Likely cause: the query is enabled before `state.kind === 'restoring'` resolves — the gate defers *rendering* but not *fetching*. Fix is probably one `enabled:` condition in the query layer. Cheap; worth doing when someone is next in that code (S1.x touches these screens).
- **The web preview's `no-cache` on `index.html` never fires for real URLs** — *(raised at S0.5, 2026-07-17; S0.4's code, found by running the preview container locally)* `Caddyfile.web-preview` sets `Cache-Control: no-cache` via `@html path *.html`, but that matcher tests the **request path**, and no founder ever requests `*.html`: they visit `/`, and shared links hit `/itineraries/<id>`. `try_files` rewrites those to `index.html` **after** the matcher has already decided, so the header is applied to exactly the one URL nobody uses (`/index.html` — verified: `no-cache` present there, absent on `/` and on `/itineraries/abc`). The Caddyfile's own comment names the symptom this was written to prevent: a founder "keeps loading yesterday's app against today's API and reports ghosts" after a redeploy — a bug that presents as backend flakiness while the browser quietly serves stale HTML. **Second finding, same cause:** the security headers (`X-Frame-Options`, `Referrer-Policy`) are also absent on the deep link, so the SPA route that *is* the shareable surface has the weakest headers. Likely fix: match on the response `Content-Type: text/html` rather than the request path, or set the header inside the rewrite. Not urgent (the preview is interim and pre-alpha; ETags still revalidate), and deliberately **not fixed at S0.5** — that story is a button and a build script; a Caddy change is S0.4's surface with its own verification needs. Cheap to fix, and it will bite exactly once, confusingly, at the worst moment.
- **The debug keystore lives in the generated `android/` tree** — *(raised at S0.5, 2026-07-17)* `mobile/android/app/debug.keystore` holds the debug key whose SHA-1 (`5e8f1606…f625`) is registered in Firebase `largata-dev` so Google sign-in works on dev builds — but `android/` is generated and gitignored (CNG), so that file is one `rm -rf android/` from being replaced by a freshly-minted key with a **different** SHA-1. The symptom would be Google sign-in breaking on every dev build, on a machine where nothing in git changed, with no error naming the cause — S0.2-class. Verified at S0.5 that `expo prebuild` *preserves* an existing one (md5 identical across a prebuild), so this is latent, not active, and a fresh clone hits it once at worst. Options when pulled: commit a project debug keystore (Android's is a well-known non-secret — `android`/`androiddebugkey`), or a config plugin that pins one from outside the tree, or document "re-register the new SHA-1" as a known clone step. Cheap to fix, invisible until it bites.
- **Email-verification enforcement — decided and built at S1.2, not before** — *(decided at S0.6 grilling, 2026-07-17)* the backend gate on `email_verified` lands with S1.2, where email invites finally give verification a job (prove you own the mailbox the invite went to) — confirming S0.2's original deferral.
