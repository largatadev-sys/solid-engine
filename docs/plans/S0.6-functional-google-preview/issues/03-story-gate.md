## Comments

**2026-07-17 — the rig cost an hour of rediscovery; it is now written down (CLAUDE.md).** None of it was this story's work: `adb` absent from the Git Bash PATH, `ANDROID_HOME` unset (Gradle says *"SDK location not found"*, which reads as a broken toolchain), `expo run:android` compiling green then failing at install with a `spawn ENOENT` buried in an ADBServer stack, Git Bash rewriting device paths (`MSYS_NO_PATHCONV=1`), an `adb reverse` that *listed* while carrying zero traffic, `INSTALL_FAILED_UPDATE_INCOMPATIBLE` from a leftover release-signed build, Metro serving `expo-router/entry.bundle` (not `index.bundle` — my 404 was a wrong probe, not a broken server), and `expo start` **silently skipping its dev server on a port clash while exiting zero**. Owner asked that this be captured rather than re-paid next story → CLAUDE.md gained a *Running the local verification rig* section with the port map: **8081 is pinned twice** (the backend's dev CORS allowlist *and* the OAuth client's registered origins), so **Metro is the port that moves**, not the preview.

**2026-07-17 — the console work landed in a phantom project first.** Documented fully in CLAUDE.md. Short version: a Firebase project may be absent from the Cloud console's picker until entered once *from* Firebase, and the console will cheerfully offer to create a same-named project (`largata-dev-502703`) — where a consent screen and an OAuth client were configured before anyone noticed. Everything was individually correct and would never have worked, with **no error to say why**: the origins simply would not apply. The durable rule: **identify Cloud projects by NUMBER (`309534715609`), never by name.**

**2026-07-17 — the collision behaved exactly as the grilling's truth table predicted, and one row got tested by accident.** The owner first signed in with Google on a *fresh* address, then tried email/password on it and got *"Email or password is incorrect."* That is not the collision — it is the truth table's **row 4** (Google-only account, tries email → ❌), and it is correct twice over: there is no password on that account, and the message is deliberately vague because `translate()` refuses to reveal which half of a credential was wrong (the S0.2 anti-enumeration decision). The collision proper needs the opposite order (password first, unverified, *then* Google), which the owner then ran end to end: eviction observed, reset recovery observed, both doorways working afterwards. **Verification enforcement stays at S1.2** as the grilling decided — and worth noting it would not have prevented the eviction anyway: that behaviour is Firebase's anti-takeover posture, not ours. What S1.2's backend gate on `email_verified` changes is that an unverified account cannot do anything meaningful in the first place, making the eviction a non-event.

**2026-07-17 — my propagation probe was wrong and the browser was right.** A bare `curl` to `/gsi/button` kept returning **400** long after the owner reported the button working. A curl lacks the browser context that endpoint expects, so its 400 said nothing about origin registration — a check that disagreed with reality, which is worse than no check. Deleted the monitor; the driver (real headless Chrome) is the only probe kept. Restates the standing rule in a new place: **verify at the layer that ships** — and when a check contradicts the running system, suspect the check.
