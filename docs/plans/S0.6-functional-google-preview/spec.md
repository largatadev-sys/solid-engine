# S0.6 — Functional Google sign-in on the founder preview · spec

**Status:** intent locked 2026-07-17 — grilling session, owner-confirmed. Immutable point-in-time intent (issue-tracker rule): if intent changes during implementation, append to `## Comments`; never rewrite this body.
**Context anchor:** Epic 0 · S0.5 spec + gate (the tri-state and the button this lands through) · S0.4's REST pivot (`firebaseWebRest.ts` — the constraint that shaped the mechanism) · ADR-006 (unchanged: `signInWithIdp` mints an ordinary Firebase ID token, so the backend contract does not move). Slice definition: `docs/design/07-epic-map.md` → S0.6, **as amended by this grilling** (two of the entry's claims are superseded below, on the record).

## The pull (what this is and why now)

Raised 2026-07-17 at S0.5's close — founder reversal of the call S0.4 made and S0.5 upheld: **the preview should work, not just look right.** S0.5's tri-state was built for exactly this landing (`'cosmetic'` → `'full'`). The grilling sharpened the slice in four ways the epic-map entry did not anticipate:

1. **Mechanism decided: the GIS-rendered button** — which supersedes the entry's "the sign-in screen does not change."
2. **Native gets Google's official button too** (owner parity ruling) — which narrows "no native changes" to "no native _flow_ changes."
3. **The account-collision policy is decided** (Firebase defaults; enforcement parked at S1.2).
4. **"Interim" is disambiguated: the code is permanent, the deployment is interim.** The real web surface (backlog) is this same codebase exported to web; everything S0.6 builds is inherited by it. This is product code that debuts on a throwaway deployment — written to product standard, not scaffolding standard.

## Goal

A founder signs into `founders.largata.com` with Google and reaches My Trips, provisioned from the **same Firebase UID** the native app would use (one identity, both doorways — ADR-006's point). Both platforms show Google's official button. Email/password remains a working doorway. The backend never learns any of this happened.

## Locked decisions

### The mechanism — GIS-rendered button (owner call, costs priced in)

- **Three options were weighed:** (a) GIS token client behind our existing button (screen unchanged, access token), (b) the GIS-rendered official button (ID token, Google-branded iframe), (c) manual OAuth redirect (no CDN script, we own `state`/nonce and lose SPA state). **Owner chose (b):** the current button styling is pre-palette placeholder anyway (visual-direction item, due before E4), so (b)'s visual-divergence cost is ~zero; founders see the genuine Google button at the moment of credential entry; it hands us an **ID token** directly; and it is the flow Google is steering toward (FedCM). The costs were accepted knowingly: screen surgery (next bullet), a control-flow inversion (bullet after), and superseding the epic map's "screen does not change" line.
- **The component seam:** `sign-in.tsx`'s inline Google `Pressable` extracts into a platform-forked **`GoogleSignInButton`** component (`.native.tsx` / `.web.tsx`). The screen changes once, structurally; the platform fork lives at the component seam like the repository's. The capability gate (`google !== 'none'`) stays in the screen, unchanged.
- **Web is push-shaped; the wiring must keep failures loud.** GIS owns the tap: the credential arrives in a callback, not from our `onPress`. The web button component routes credential → repository (`signInWithIdp`) → and **must** drive the screen's existing `busy`/`message` machinery, so a post-popup failure surfaces exactly like any other auth error. A silent nothing after the Google popup is the dead-click S0.5 refused to ship, one layer deeper.
- **Button-only — no One Tap.** No unprompted overlay on page load, no `auto_select`. A click opens Google's account chooser (full multi-account menu — verified in grilling this loses nothing for multi-account browsers; One Tap is the option that _narrows_ to a suggested default). One Tap is conversion tooling for consumer funnels; the preview has five users who were told to click the button. This also keeps the web surface's behavior a strict subset of the shipping app's.

### Native gets the official button too (owner parity ruling)

- `.native.tsx` renders **`GoogleSigninButton`** from `@react-native-google-signin/google-signin` — already a dependency; no new module, no prebuild change, no config plugin, no SHA-1 implication.
- **The flow is untouched:** native stays pull-shaped — `onPress` → `run('google', …)` → `authRepository.signInWithGoogle()` → account picker. Only the resting button's pixels change. The chain proven on the owner's physical phone at S0.5 is not disturbed.
- **Standing UI principle recorded (owner, 2026-07-17):** _"web and mobile identical as much as possible, except native gestures and native-only functions."_ It decided this question and should decide future ones; the visual-direction backlog item inherits it.
- **Evidence bar:** emulator smoke (button renders; tap opens the account picker). **No release-APK re-proof** — signing, SHA-1, and the token flow are unchanged, so the S0.5 device evidence stands.
- **Expectation set now:** the GIS web button and the native `GoogleSigninButton` are both Google-branded but not pixel-identical (different rendering stacks). "As much as possible" is satisfied; "identical" won't be.

### The web doorway (code)

- **`firebaseWebRest.ts` gains `signInWithIdp`:** POST `accounts:signInWithIdp` with the GIS ID token (`postBody: id_token=…&providerId=google.com`, `returnSecureToken: true`, a `requestUri`), session stored via the same `storeFromAuthResponse` path as password sign-in — persistence, refresh, and listeners reused wholesale. That reuse is why this story is small.
- **The web install path — and the latent crash it defuses:** `googleSignInConfig.ts` imports `configureGoogleSignIn` from the platform-forked repository, but `authRepository.web.ts` **does not export it** — undefined-but-uncalled today only because the `'full'` gate keeps `_layout.tsx` from calling it (and typecheck resolves the native file, so it never noticed). Flipping web to `'full'` without a web implementation is a startup crash. S0.6 gives `.web.ts` its own `configureGoogleSignIn`: store the client id, load the GIS script, initialize button-only. The shared gate in `_layout.tsx` and the env-var read in `googleSignInConfig.ts` don't change — the gate stays, what it gates forks (exactly what `authContract.ts`'s corrected comment promised).
- **`authCapabilities.google`: web `'cosmetic'` → `'full'`.** `'full'` means "a doorway exists," never "the native SDK's doorway exists."
- **Web `signInWithGoogle()` becomes unreachable-by-design** (GIS owns the tap; the contract member must still exist). It must stay honest — throw an `AuthError` naming itself a wiring bug if ever reached, never a silent no-op. Exact shape is the implementer's call.
- **`.web.ts`'s "interim scaffolding" framing is softened** in its doc comment: the deployment is interim; this code is the permanent web doorway the real web surface inherits.

### Identity & the collision (Firebase defaults, no enforcement)

- **One account per email; the truth table was walked in grilling.** Same email → same UID → same Traveler — ADR-006 holds with zero work. Email sign-in against a Google-only account already fails correctly and vaguely; sign-up against any existing email already fails with "email already in use."
- **The sharp edge, accepted:** Google sign-in **evicts an unverified password credential** (Firebase's anti-takeover posture — an unverified password claim loses to a verified Google sign-in). Recovery is built into our UI: "Forgot password" re-adds a password _and_ verifies the email, after which both doorways coexist. **The AC is therefore re-scoped: "email/password still works _as a doorway_"** — not "any pre-existing password survives Google sign-in."
- **Observed, not just cited:** the container run exercises the collision once (unverified password account → Google sign-in → old password fails → reset → both work), so the story record shows what Firebase _did_, not what was recalled about it.
- **Verification enforcement stays parked at S1.2** (backlog line added this story). Real enforcement is a backend gate on `email_verified` — a backend change, out of S0.6's scope — and S0.2 already assigned the decision to S1.2, where invites give verification a job. Client-only enforcement was rejected as theater.
- **Demo note to founders:** _"if your old password stops working after you've used Google, tap Forgot password to set a new one — then both work."_

### Config & console (owner prerequisites — do early; origin propagation can lag hours)

- **Reuse the Firebase auto-created web OAuth client** — the value already in `EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID` (native uses it for the token exchange; it _is_ a web client id). No new client, no new env var. A dedicated preview client was rejected: ceremony that outlives its purpose.
- **Owner console steps:** (1) Google Cloud console, `largata-dev` → the web OAuth client → **authorized JavaScript origins**: `https://founders.largata.com`, `http://localhost`, `http://localhost:8081` (bare-host and host:port are distinct entries; localhost is the one plain-http host Google allows). (2) Firebase → Authentication → Settings → **Authorized domains** → add `founders.largata.com` (localhost is pre-authorized).
- **`Dockerfile.web-preview` gains `ARG`/`ENV EXPO_PUBLIC_GOOGLE_WEB_CLIENT_ID`**; the Railway preview service gets the build arg; `mobile/.env.example` gains the line. The S0.4 inlining gotcha applies: **direct static member access only**, and the container verification greps the served bundle for the baked value before trusting it.
- **Decommission list grows:** the OAuth origins and the Firebase authorized domain join "delete the Railway service" as preview-teardown steps (recorded so the cleanup list stays true).

### Verification (the layer that ships)

- **Preview container, before the merge is proposed** (standing rule): built with real build args, served at **`localhost:8081`** (the origin docker-compose's CORS allowlist already permits — the port S0.5's harness got wrong), against the **full local stack** (Spring + Postgres 18 + MinIO, fresh DB). The real Google round-trip: GIS popup → ID token → `signInWithIdp` → API calls → backend logs `Traveler provisioned`, DB 0 → 1. Plus: email round-trip · the collision scenario · no One Tap overlay on a signed-out load · bundle grep (client id baked, `'full'` declared).
- **Post-merge, closes the gate:** a founder (or the owner in a cold browser) signs into `founders.largata.com` with Google and reaches My Trips.
- The local backend trusts `largata-dev` (same as the deployed preview), so the token chain is identical on both rungs — the container proof is real proof.

## Comments

**2026-07-17 — intent changed once during the build: "Google's official button" became "Google's official *mark*, on our button".** This spec locked *"`.native.tsx` renders `GoogleSigninButton` from `@react-native-google-signin`"* — the founder's parity ruling, taken literally. Seen running on the emulator, it was wrong: the library renders Google's **older** button spec (square corners, heavy drop shadow, "Sign in with Google") directly above our flat pill, and it exposes only `Size`/`Color`, so it could not be tuned. **It also inverted the ruling it came from** — GIS renders a *flat pill* reading *"Continue with Google"* on web, so the official-component-on-both approach made the two surfaces **less** alike than before. The ruling ("web and mobile identical as much as possible") is what decided the correction: native now draws our own pill carrying Google's unmodified mark (`assets/google-g-logo.png`, their published asset — their branding terms permit a custom button on exactly those conditions), matching the web button's shape, wording and flatness. **The flow is untouched as specced** and was re-proven on the emulator afterwards. Side effect worth noting: the busy spinner ticket 02 had accepted losing is **back** (a `busy` prop on the button contract), and `GoogleSignInButton.native.tsx` came back **off** `layering.test.ts`'s SDK allowlist — it no longer imports the SDK at all.

**2026-07-17 — the ACs' proof, and one limit stated honestly.** All owner-run against the preview container on the fresh-DB local stack: Google round-trip to My Trips · email round-trip · the collision sequence (unverified password → Google → old password fails → reset → both doorways work). **The claim no screen could show, checked at the database:** 3 travelers, 3 distinct Firebase UIDs, 3 `Traveler provisioned` events — provisioning fires once per *new* UID, so a second identity for the collision address would have left a 4th of each. None. The UID survived the eviction and the Traveler was reused: ADR-006's "one identity, both doorways" holding in practice, and the reason this story changed no backend code. *Limit:* the log records UIDs, never emails (P3), so which UID was the collision address rests on the owner's account of the steps; the counts corroborate rather than independently prove it.

**2026-07-17 — two checks that lied, both recorded in CLAUDE.md.** (1) A `curl`-based watcher on `/gsi/button` reported **400 indefinitely** while the button was demonstrably working in a browser — a bare request lacks the context that endpoint expects. A check that contradicts the running system is worse than no check; it was deleted and the CLAUDE.md advice recommending it was corrected to "load the real page". (2) The first attempt at the Google mark used an **SVG data URI**, which would have shipped **blank on Android** (RN's `<Image>` uses Fresco, which has no SVG decoder) while rendering perfectly on web — the S0.2 `getTokens()` shape again, caught before landing. Neither cost the story; both are the kind of thing that costs the *next* one.
