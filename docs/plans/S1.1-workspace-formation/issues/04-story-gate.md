## Comments

**2026-07-17 ‚Äî two verification errors on the way to closing this, both worth recording because both cost real time and neither was in the code.**

**(1) A probe that could not discriminate, trusted for a moment anyway.** To detect whether the deploy had landed I created an itinerary against deployed dev and immediately viewed it, reasoning that only the new build could serve the view. Wrong: *both* builds return 200 ‚Äî the old one answers from `owner_id`, the new one from a membership row. Same response, opposite mechanisms. (The deploy log later showed the probe *had* hit the new build, so it was right by accident ‚Äî right answer, invalid method.) This is the `/gsi/button` lesson in a new costume: **a check whose two outcomes are indistinguishable is not a check.** The genuinely discriminating signals were `flyway_schema_history` and the deploy log; neither is reachable from the API, which is the honest reason AC 6 needed the database.

**(2) "No rows" from the wrong database looks exactly like "the deploy never happened."** The first run of the AC-6 query returned nothing, and both agent and owner read it as *the migrations have not run* ‚Äî a false alarm that sent the investigation to CI and `gh auth`. The deployed backend uses **`postgres.railway.internal:5432/railway`** ‚Äî database **`railway`**, not a name anyone would guess ‚Äî and the SQL had been handed over without naming it. **Any query against a deployed rung must name the database it belongs to**, or its null result is ambiguous between "the answer is no" and "you asked the wrong server". Same family as ADR-006's environment identity and S0.6's phantom Cloud project: the thing you are talking to must be identified, never assumed.

**2026-07-17 ‚Äî one piece of junk data left on dev, knowingly.** The probe account (`s11-dev-‚Ä¶@largata.test`) owns an itinerary titled *"S1.1 deploy probe"* ‚Äî it is the +1 in every count above (**4 founder trips + 1 probe**). It cannot be removed cleanly: there is no delete endpoint until S1.9, and deleting rows by hand on a deployed rung to tidy up is not a thing this workflow does casually. It costs nothing (dev is the shared preview, reseeded at the developer's discretion) and it disappears at the next reseed. Recorded so the next person reading these counts is not puzzled by a fifth trip nobody remembers planning ‚Äî and so the "5" is not mistaken for five founder trips.
