**2026-07-17 — the full local stack, and the false green it caught.** The owner pointed out that 06b §17 / Artifact 04 §83–89 already say what S0.5 had been ignoring: **feature branches are tested on the full-stack local Docker instance** (Spring + Postgres + MinIO, *fresh DB every redeploy*), and `dev` is the long-lived **preview**, not a test rung. Everything in this story had been driven against the deployed `dev` backend instead — which wrote test data into the environment founders look at (an "S05 / Kyoto" trip is in the `dev` DB), and let pre-existing rows (an S0.3-era "Aussie" trip) stand in for proof the flow works from nothing. The topology's ephemerality is *the* point: a fresh DB is what makes the result deterministic. `docker-compose.yml` was there the whole time; the rule was in the design package the whole time. It got lost in execution, not in translation.

Run properly, it immediately caught a false green worth recording: the flow appeared to reach **My Trips**, which read as success — but the `traveler` table was still **empty** and the backend had logged nothing but health checks. Sign-in had flipped the *auth state* (a Firebase cloud call) and the router did its job; the trips fetch had failed silently behind an honest error banner. **Routing is not a round-trip**, and only looking at the database and the network log distinguished them. The cause was a **CORS 403 on the preflight** — my harness's fault, not the code's: I invented port 8101, and `LARGATA_CORS_ALLOWED_ORIGINS` (docker-compose:61) allows 8081/8082. Restarted with the origin added → preflight **200** with the right `Access-Control-Allow-Origin` (the S0.4 chain working as designed), then the real result: **`GET /v1/itineraries` → 200**, `Traveler provisioned: id=019f6bb2…` + `event=traveler_signed_up` in the log, DB **0 → 1 travelers**, "No trips yet" rendered honestly.
